# テストサイト保護メモ

## 目的

- 開発初期の `socia_maker` を、しばらく友人向けのクローズドテストとして公開する。
- アプリ内の本実装 `auth/member/public-share/license` が整う前に、サイト全体へ暫定の入場制限をかける。

## 現在の前提

- 現在のアプリには、サイト全体へパスワードをかける仕組みはまだない。
- 共有URLの仕組みは作り始めているが、これは「プロジェクト共有」の土台であり、サイト全体の入場制限ではない。
- そのため、テストサイト保護はアプリ内実装ではなく Cloudflare 側で行うのが現実的。

## 当面の推奨方針

- `Cloudflare Access` を使って Pages サイト全体にアクセス制限をかける。
- アプリ側では独自ログインを急いで作らない。
- 友人テストの期間中は「URLを知っているだけでは入れない」状態を先に作る。

## 採用理由

- 実装コストが低い。
- 既存の SPA や API を大きく壊さずに導入できる。
- テスト終了後に解除しやすい。
- 本番用の auth 設計を急いで確定しなくてよい。

## 守る対象

- Pages の公開URL全体
- `home`, `gacha`, `story`, `collection`, `editor`
- `/api/*` を含む Functions

## 推奨する制御レベル

### フェーズ1: クローズドテスト

- サイト全体を Access 配下に置く。
- 許可された人だけ入れる。
- 想定利用者は作者本人と友人テスターのみ。

### フェーズ2: 限定公開テスト

- 必要なら閲覧者を少し増やす。
- ただしアプリ内公開共有はまだ主導線にしない。

### フェーズ3: 将来の正式公開

- アプリ内の `collab share` と `public share` を正式機能として運用する。
- 必要に応じて Access 保護を外す。

## Cloudflare Access の運用イメージ

### 最小運用

- Pages プロジェクト全体に Access policy を付ける。
- テスターごとに許可するメールアドレスを登録する。

### 代替運用

- 共有用の共通ワンタイムコードや簡易認証を使う運用もありうる。
- ただし、まずはメール許可型のほうが管理しやすい。

## テスト期間中のおすすめ運用

- テスター一覧を先に決める。
- 誰に入場を許可したかをメモする。
- 毎週または節目で参加者を見直す。
- 外部へ広がりそうなら Access 設定を優先して締める。

## アプリ側で今やらないこと

- 独自の会員登録
- 独自のログイン画面
- パスワード再設定
- 本格的な権限管理

これらは将来の `auth/member/public-share/license` 実装で扱う。

## 実装・運用の切り分け

### Cloudflare 側で担当するもの

- サイトへ入れる人の制御
- テスト期間中の入場制限
- 一時的な保護

### アプリ側で担当するもの

- project 単位の共有
- 共同編集URL
- 公開プレイ専用URL
- 将来の owner / member / viewer 権限

## 友人テスト期間の判断基準

次の状態になったら、アプリ内公開機能へ段階的に寄せていく。

- クラッシュや保存消失が減った
- 編集導線が迷いにくくなった
- home / gacha / story の最低限のループが安定した
- 共有URL仕様が固まった

## 次の具体アクション

1. Cloudflare Access で Pages サイト全体を保護する
2. 友人テスターを 3〜5 人程度に絞る
3. 触ってほしい観点をチェックリスト化する
4. フィードバックを受けて優先順位を更新する
