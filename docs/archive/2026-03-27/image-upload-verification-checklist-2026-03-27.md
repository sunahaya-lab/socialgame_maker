# 画像アップロード確認チェック手順

## 目的

- `WebP` 正規化アップロードが実際に動いているかを、公開テスト前に確認する。
- `frontend -> Functions -> R2 -> D1 -> 再表示` の一連の流れを最低限チェックする。

## 前提

以下が揃っていること。

- `0012_asset_webp_normalization.sql` が remote D1 に適用済み
- `SOCIA_ASSETS` が `wrangler.toml` に設定済み
- `socia-maker-assets`
- `socia-maker-assets-preview`
  の R2 バケットが作成済み

---

## 確認したいこと

最低限、次を確認する。

- 画像選択後にエラーで止まらない
- 保存後に画像が表示される
- `assets` テーブルにメタデータが入る
- `R2` に本体が入る
- `/api/assets-content?id=...` で画像が読める

---

## テスト対象

最初は次の2種類で十分。

- ベースキャラ画像
- カード画像

---

## 手順 1: ベースキャラ画像アップロード確認

1. アプリを開く
2. editor からベースキャラ編集を開く
3. 新規ベースキャラを作る
4. 名前を入れる
5. 画像を1枚選ぶ
6. 保存する

### その場で見ること

- 保存でクラッシュしない
- 画像が保存後にそのまま見える
- トーストやエラー表示が不自然でない

---

## 手順 2: カード画像アップロード確認

1. editor からカード編集を開く
2. 新規カードを作る
3. カード名を入れる
4. rarity を選ぶ
5. 戦闘プリセットを選ぶ
6. 画像を1枚選ぶ
7. 保存する

### その場で見ること

- 保存でクラッシュしない
- カード一覧や編集画面で画像が見える
- データURLではなく、保存後の URL 参照で見えている

---

## 手順 3: 表示URLの確認

保存後、画像 URL が次の形式になっているかを見る。

- `/api/assets-content?id=...`

### 見るポイント

- 相対URLとして保存されている
- 表示崩れがない
- 同じ URL をブラウザで直接開いて画像が出る

---

## 手順 4: D1 メタデータ確認

`assets` テーブルで、対象 row を確認する。

見たい項目:

- `id`
- `project_id`
- `owner_user_id`
- `kind`
- `usage_type`
- `mime_type`
- `stored_format`
- `quota_bytes`
- `normalization_status`
- `width`
- `height`
- `r2_key`

### 成功条件

- `mime_type = image/webp`
- `stored_format = webp`
- `normalization_status = ready`
- `r2_key` が入っている
- `owner_user_id` が入っている

---

## 手順 5: R2 本体確認

R2 側で対象ファイルが保存されているかを確認する。

見たいこと:

- 対象キーが存在する
- `projects/{projectId}/users/{userId}/assets/...` の形になっている
- 1つのアップロードにつき 1 本体だけ入っている

### 成功条件

- アップロードした画像に対応する object がある
- content-type が `image/webp` で扱える

---

## 手順 6: 再表示確認

保存後に、画面再読込を行う。

### 見ること

- ベースキャラ画像が残っている
- カード画像が残っている
- 画像が消えない
- 画像表示が data URL 依存でない

---

## 手順 7: フォールバック確認

可能なら、R2 か API を一時的に失敗させた時に、ローカルフォールバックが出るかも確認する。

### 見ること

- 保存時に完全クラッシュしない
- トーストが出る
- 一時的にローカル画像を使う挙動になる

これは必須ではないが、余裕があれば確認するとよい

---

## 成功条件

最低限、次が通れば十分。

- ベースキャラ画像アップロード成功
- カード画像アップロード成功
- `/api/assets-content?id=...` で表示可能
- `assets` テーブルに記録あり
- `R2` に object あり
- リロード後も画像が表示される

---

## 失敗時の切り分け

### 1. 保存直後に 503

見る場所:

- `SOCIA_ASSETS` binding
- `wrangler.toml`
- Pages 側デプロイ反映

### 2. 画像は保存できたが表示されない

見る場所:

- `sanitizeImageSource`
- `/api/assets-content`
- `mime_type`
- `r2_key`

### 3. D1 にはあるが R2 にない

見る場所:

- upload API の途中失敗
- bucket binding
- R2 側 object 保存成否

### 4. R2 にはあるが D1 にない

見る場所:

- `insertAssetRecord`
- migration 適用漏れ
- D1 書き込み失敗ログ

---

## 次のアクション

この確認が通ったら、次は

- `カード -> ガチャ -> 引く -> 編成 -> 戦闘`

の最短導線を 1 周確認する。
