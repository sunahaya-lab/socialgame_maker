# クローズドテスト公開チェックリスト

## 目的

- `socia_maker` を友人向けに実際に触ってもらう前に、公開テストの最低限の準備を揃える。
- まだアプリ内の auth / member / public-share / license が完成していない前提で、安全にテスト公開へ進む。

## 基本方針

- 先に **サイト全体の入口を閉じる**
- 次に **データ基盤を揃える**
- 最後に **アプリ内の最低限動作を確認する**

---

## Phase 1: サイト全体の保護

### 1. Cloudflare Access を有効化する

- Pages サイト全体を Access 配下に置く
- `/`
- `/api/*`
- すべての SPA 画面
  - `home`
  - `gacha`
  - `story`
  - `collection`
  - `editor`

### 2. テスターを絞る

- 最初は 3〜5 人程度にする
- 許可対象をメモしておく
- 一時的なクローズドテストであることを前提にする

### 3. 公開先 URL を決める

- クローズドテスト用 URL を1つ決める
- できれば本番導線とは分ける
- テスターにはその URL のみ共有する

---

## Phase 2: インフラとデータ基盤

### 4. remote migration を適用する

優先度が高い migration:

- `0009_share_access_tables.sql`
- `0010_user_billing_entitlements.sql`
- `0012_asset_webp_normalization.sql`

必要に応じて:

- `0007_player_home_preferences.sql`
- `0008_player_currency_balances.sql`
- `0011_player_event_progress.sql`

### 5. R2 バケットを用意する

- `SOCIA_ASSETS` 用の R2 バケットを作る
- preview / production の運用方針を決める

### 6. `wrangler.toml` に R2 バインディングを入れる

対象:

- `SOCIA_ASSETS`

補足:

- 現在の `WebP` 正規化アップロード実装は R2 前提
- 未設定のままだと画像アップロード API は `503` を返す

### 7. 必要なシークレットを確認する

最低限確認したいもの:

- `SOCIA_ADMIN_SECRET` の有無
- Access 側の制御設定
- D1 / KV / R2 の binding 名の一致

---

## Phase 3: アプリの最低限動作確認

### 8. カード作成ができる

- カード名
- 画像
- rarity
- 戦闘プリセット

この最小入力で保存できること

### 9. ガチャ作成ができる

- ガチャ名
- ガチャテンプレート
- 排出対象カード

この最小入力で保存できること

### 10. ガチャを引ける

- 1回引き
- 10回引き
- 結果が表示される
- 所持カードに反映される

### 11. 編成できる

- 5人編成が開ける
- リーダー / サブリーダーが分かる
- 所持カードを編成に入れられる

### 12. テスト戦闘に行ける

- 編成画面から `テスト戦闘` に入れる
- `単体敵`
- `複数敵`
- `ボス敵`
  のいずれかと戦える

### 13. 戦闘がそれっぽく見える

- 通常攻撃で画像が動く
- スキルで立ち絵強調が出る
- 必殺技でカットインが出る
- オートが動く
- セミオートでスキルを押せる

### 14. ホームが最低限成立している

- ホームが表示される
- ガチャ / ストーリー / コレクションへの導線が見える
- 表示崩れや重複表示がない

### 15. 画像アップロードが動く

- ベースキャラ画像
- カード画像

について、

- `WebP` 正規化アップロードが通る
- 保存後の表示に使える
- R2 / D1 の両方に反映される

---

## Phase 4: テスト向け運用準備

### 16. テスター向け説明を用意する

最低限伝えること:

- これはクローズドテスト版である
- データが壊れる可能性がある
- どこを見てほしいか
  - カード作成
  - ガチャ
  - 編成
  - 戦闘
  - ホーム

### 17. フィードバックの受け皿を決める

- Discord
- DM
- フォーム
- ドキュメント

のどれで受けるかを決める

### 18. テスト観点を絞る

最初は全部見てもらわない

推奨:

- 画面が分かるか
- 保存が通るか
- 戦闘がそれっぽいか
- ホームが触って楽しいか

---

## Phase 5: アプリ内公開共有の確認

### 19. `public share` は第二段階で確認する

- サイト全体保護より先にやらない
- まず Access で入口を制御する
- その後、必要なら project 単位の公開URLを試す

### 20. `public share` 確認時に見ること

- Publish 権限付き project でのみ有効か
- 発行できるか
- 無効 token が `410` 相当になるか
- `play_only` で編集不能になっているか

---

## 公開前の最終確認

- Access でサイト全体が閉じている
- migration が揃っている
- `SOCIA_ASSETS` が設定済み
- 画像アップロードが動く
- カード → ガチャ → 引く → 編成 → 戦闘 が通る
- 明らかなクラッシュがない

---

## 最初にやるべきこと

順番を1行で言うと、これです。

1. Cloudflare Access でサイト全体を閉じる
2. remote migration を入れる
3. `SOCIA_ASSETS` を設定する
4. 最短導線を自分で1周する
5. 少人数テスターへ渡す

---

## 補足

- 今の段階では、アプリ内の公開URL機能より **外側の保護** を優先する
- `public share` は将来の正式機能であり、クローズドテストの入口制御とは役割が違う
- まずは「ちゃんと閉じて、ちゃんと一周動く」ことが大事
